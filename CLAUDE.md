# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

**Based Answers** — a citation-grounded QA web app (it lives under `~/.config/opencode/skills/` for historical reasons, but the agent it drives is `pi`, not opencode): questions are typed into a browser question bar; per question, an LLM agent produces a YAML answer where every claim cites a verbatim source quote with page number, verified by deterministic and LLM-based checkers. The run's progress (draft answer, per-claim check status, live output of all three agents) streams to the browser and is persisted, so past runs stay fully viewable across server restarts.

## Running

Python dependencies (PyMuPDF, PyYAML, Jinja2, Flask, flask-sock, simple-websocket) and the `pi` agent binary come from the Nix flake. Nix appears only at the entry point: nothing in the source shells out to `nix develop`, and the scripts do *not* re-enter the flake themselves — they need an interpreter that already has the dependencies, which is what the canonical invocation provides:

```sh
# Start the web app (run from a working directory containing the PDFs, not from the skill dir)
nix develop "path:/Users/robin/.config/opencode/skills/citation-grounded-qa" -c python3 /Users/robin/.config/opencode/skills/citation-grounded-qa/based-answers.py
```

This indexes the CWD's `*.pdf`, starts Flask on an auto-assigned port (`--port` pins one, `--no-open` skips the browser), and serves until Ctrl-C. Questions are asked in the UI, never as CLI arguments.

**Credentials**: the LLM is reached through pi, which needs `DEEPSEEK_API_KEY` in its process environment. `api_key()` takes it from the environment if set, else from the macOS Keychain generic password under service `deepseek` (`BA_KEYCHAIN_SERVICE` overrides). A global `pi /login` does **not** apply — `PI_CODING_AGENT_DIR` is redirected (see below), so pi never reads `~/.pi/agent/auth.json`. Startup prints which source was used, never the key. The model is `deepseek/deepseek-v4-flash` (`BA_PI_MODEL` overrides).

**A model swap is not just `BA_PI_MODEL`.** Any replacement must support **tool calls** — a searcher that cannot call `pdf_search` produces nothing — and pi's bundled model table is stale enough that being *in* it proves nothing: `openrouter/free` is listed there and 404s with "This model is unavailable for free". pi passes unknown slugs straight through to the provider, so the table is neither necessary nor sufficient; test the slug against the real tools before trusting it.

**A rejected model looks like a bad answer, not an error.** When the provider 404s or 429s, the agent emits no assistant message, so the round is indistinguishable from an agent that simply wrote no YAML. **Zero tokens is the signature of a transport/model/auth failure** — check the slug and the key before touching prompts. `searcher_produced_nothing()` now reads that signature off the ledger after round 1 (`calls > 0` with `total == 0`) and raises, so the run ends as `error` naming `BA_PI_MODEL` and the key, instead of burning all 5 rounds and reporting `exhausted` — which reads as "tried and could not answer". Only round 1 is checked: the ledger is cumulative, so a later round reading zero implies round 1 did too.

Tests (stdlib unittest, in `tests/`; `tests/support.py` loads the dash-named scripts as modules; `tests/fixtures/` holds the RP2040 datasheet + real answer YAMLs driving the end-to-end tests):

```sh
nix develop "path:." -c python3 -m unittest discover -s tests -t .
```

There are no linters.

All scripts are **working-directory-relative**: they read/write `answers/`, `indexed-pdfs/` (extracted-text cache, keyed by PDF basename), `.based-answers/` (all pi state), and `citation-qa.db` in the CWD. Run the app from the directory where the question artifacts should live.

## Architecture

`based-answers.py` is the entry point: it indexes PDFs, then constructs `live-server.py`'s `PipelineServer` (Flask + flask-sock served by werkzeug in a daemon thread) and blocks. Each submitted question spawns its own worker thread (`run_worker`), so multiple runs proceed in parallel.

**Nothing is installed into `$HOME`.** Earlier versions copied agent definitions into `~/.config/opencode/agents/` and generated tool wrappers into `~/.config/opencode/tools/` on every start; none of that exists now. The agent prompts are read from this repo and passed as `--system-prompt`, and the tools are checked-in `.ts` files handed to pi by absolute path.

Per run (run-id = question slug, `-N` suffixed on re-asks):

1. **The agent transport** (`pi_rpc.py`) — `PiSession` runs `pi --mode rpc`, a JSONL protocol on stdin/stdout, and drives it as a deterministic subroutine: `prompt()` sends a message and blocks until the run is over, then control returns to the pipeline. `build_command()` fixes the isolation, and every flag there is load-bearing:
   - `--no-extensions --no-approve` — nothing is discovered from `~/.pi` or a project `.pi`; only explicit `-e` paths load (pi's own help confirms `-e` still works under `--no-extensions`).
   - `--no-builtin-tools` + `--tools` — the agent gets exactly the pipeline's tools and no bash/edit/write/web. **pi does not validate `--tools`**: a name no extension registers is accepted silently, leaving the agent with no tools and failing several rounds later as a vague "cannot search". `tests/test_tool_names.py` pins `SEARCH_TOOLS`, `TOOL_EXTENSIONS` and the `registerTool()` names in `tools/*.ts` to each other so drift fails at test time.
   - `--system-prompt` — `citation-searcher.md` / `coherence-checker.md` read straight from this repo.
   - `PI_CODING_AGENT_DIR=.based-answers/pi` — settings, auth, trust and sessions all relocate under the CWD, so a run can neither read nor mutate global pi state.

   **Settling is version-dependent.** pi ≥ 0.80 emits `agent_settled` once it will not continue automatically; pi 0.79 (the version in nixpkgs) has no such event and its `agent_end` is terminal with no `willRetry`. So `agent_end` arms a settle `_SETTLE_GRACE` seconds out rather than ending the round: on newer pi the `agent_settled` that follows wins the race, on 0.79 the grace expires. Any event in `_WORK_RESUMED` disarms it, so retries and queued continuations are still waited out on both. **When something looks wrong, the authoritative reference is the docs in the nix store** (`$(dirname $(readlink -f $(which pi)))/../lib/node_modules/pi-monorepo/docs/`), not the latest npm tarball — nixpkgs lags, and both bugs found in the first live run came from reading 0.81 docs against a 0.79 binary.

   The three tools live in `tools/*.ts` as checked-in pi extensions. They need no `package.json` or `node_modules` — only pi's own types and node builtins — and take `BA_PYTHON` (this process's already-resolved interpreter) and `BA_SKILL_DIR` from the environment, so they invoke the Python scripts directly whatever provided that interpreter.
2. **Search loop** (max `MAX_ROUNDS` = 5 rounds) — `search_loop()` opens **one `PiSession` for the whole run** and closes it at the end. Round 1 prompts with the context written to `answers/<slug>-context.md`; later rounds prompt the *same* process with that round's feedback, which is the same conversation. There is no session id to discover, pass, or lose: the previous `opencode session list` diffing, title matching, `_claimed_sessions` bookkeeping and fresh-session fallback are all gone.

   **The context file carries only what the session doesn't already have, and only what the agent can act on** (`write_context`): the question, and one line per PDF (bare filename + page count — the tools run in the CWD, so no absolute path is needed). It deliberately does *not* repeat `citation-searcher.md`'s body, which pi already loads via `--system-prompt`. "File" is a slight misnomer: `answers/<slug>-context.md` is written so a human can see what the agent was given, then read straight back and sent as the round-1 *message*. Nothing is attached and the agent never opens anything.

   **It used to offer a ranked shortlist of related past answers, and that was an instruction to do the impossible.** `relevant_answer_files()` scored every `answers/*.yml` by content-word overlap and listed the top 10 by filename, with `citation-searcher.md` telling the agent to read the relevant ones — but `SEARCH_TOOLS` is `pdf_search`, `verify_citations`, `write_answer` under `--no-builtin-tools`, so there has never been a way to open one. The shortlist, its scorer (`content_words`/`STOPWORDS`) and both prompt rules pointing at it are gone. Reuse across runs was the goal and is still worth having; the way to get it is to inline the shortlisted answers' *claims* into the context message, not their filenames.

   `build_feedback_message()` sends **only the round just finished** — the one `PiSession` holds every earlier round verbatim — and is the only caller of `group_feedback_by_round()`. `write_context()` renders no feedback at all: with the fresh-session fallback gone it is only ever used for round 1.
3. **Three checkers per round**, each failure appended to `rounds` feedback and triggering a retry:
   - **Deterministic** (`verify-citations.py`): a citation with `spans` is re-resolved through `check_spans()` and must equal the lines it cites (see the answer contract below — this is now the normal path and passes by construction). A legacy `text`-only citation must be at least 200 chars (`MIN_CITATION_CHARS`) and appear on the stated page of the cached extraction, tried through a ladder of increasingly lenient matches (exact → whitespace-normalized → case-insensitive → hyphen-stripped → arrow-normalized → punctuation-stripped); a quote spanning a page break passes when ≥20 chars of it match the stated page and the remainder matches the adjacent page (headers/footers sit between the halves, so joined page text can't match). On failure it reports closest-match suggestions and a found-on-page-X hint when the quote lives on a different page.

**A failure that describes the problem instead of handing back the fix costs rounds.** Run `…-14` spent four verify calls on one snippet whose only feedback was "copy the snippet verbatim from pdf_search's output" — true, and useless, since the agent believed it had. That is the failure the span-based contract now makes unauthorable; the machinery below survives for legacy `text` citations, and its lesson is the reason `resolve-answer.py` reports every unresolvable citation at once instead of one per round. Two failures return pasteable text between `>>>`/`<<<` markers. `_verbatim_help()` (both `search_result` branches) either names the page the text is actually on, or returns the closest text on the stated page verbatim, up to `SUGGESTION_CHARS`; since the match ladder is already lenient about whitespace, case, typography and punctuation, a failure here means the *wording* diverges, so the message says not to retype it. `_expand_verbatim()` handles the too-short citation: it locates the quote via `pdf_search.normalize_for_match()`'s index map and widens it in the original text to `MIN_CITATION_CHARS + SHORT_CITATION_MARGIN`, never past the chunk it lives in. A test pins the round trip — pasting back what the verifier offered must pass. It also emits **advisories**: lines prefixed `ADVISORY: ` that never affect the exit code. The only one today flags a claim whose entire evidence is ≥3 literal `search_result` citations that each found nothing (`guess_list_advisory()`) — the "I guessed N names and none existed" shape that spun rounds 1–3 of `…pin-directions-of-a-pio-10`, where the semantic checker answered each round by naming an N+1th spelling, so the searcher patched the list instead of switching to an enumeration. Fixed wording delivered the first time the shape appears beats a better objection later. It is advisory because it reads only the shape and never what the queries say, so it cannot tell a genuinely exhaustive handful of probes from a hopeful one; the checker, which can, keeps the final say. `deterministic_advisories()` in `based-answers.py` slices those lines out of the verifier output and appends them to whatever feedback the round produces — a round that passes verification and then fails semantically sends only the checker's objection, so without that the advisory would never reach the agent. Its table lists **only failing citations** plus the totals — that output is both the retry feedback and the browser's verifier-output block (which `run.html` renders only on failure), and a passing row gives the agent nothing to act on; `--show-passes` restores the full table for humans.
   - **Semantic** (per claim in order, with all preceding claims passed as trusted premises; stops at the first failing claim and returns to the searcher without checking the rest) and **coherence** (all claims joined into one paragraph): rubric prompts sent by `run_checker()`, which builds a **fresh, tool-less `PiSession` per check** (`tools=[]` means `--tools` is omitted entirely, so the checker judges the text it is given and cannot go consult the sources). Pass requires "PASS" and no "FAIL" in the output — which is why `stream_prompt()` returns only the `text_delta` stream: the model is a reasoning model, and a checker that weighs "this could FAIL if…" before answering PASS would otherwise be read as a failure. The rubrics live inline in `based-answers.py`.

     **Passed claims latch for the whole run.** `_search_rounds()` owns a `passed_claims: set[str]` and threads it into every `run_semantic_checkers()` call; a claim unchanged since a round where it passed is skipped (emitting `claim-status` `pass` plus a "SKIPPED" stream line) instead of re-judged. The checker is an LLM and is not deterministic at the margin — an untouched claim that passed in round 3 can fail in round 5 and end an otherwise-succeeding run purely on a re-roll, which is what killed `…pin-directions-of-a-pio-6`. A second verdict on identical input is a new sample, not new evidence. The key is `claim_key()`: the claim text **and** its citations, JSON-serialized with `sort_keys=True` so a rewrite that only reorders mapping keys doesn't read as a change. Citations are in the key because they are the evidence the verdict was about — reworded claim, new quote, swapped query or enumeration all go back through the checker. That is a deliberately narrow latch: in the run above the citations did change each round, so it would not by itself have saved that one.

     A checker that fails to run reports the error **into its own output** rather than returning empty — empty contains no "FAIL" and would be read as a pass, which is the worst failure mode this pipeline has. `run_checker()` must also build its session with `pi_env()`: the API key is read from the Keychain, so it is absent from `os.environ` and is *not* inherited by a subprocess. A hand-written env dict there leaves every check unauthenticated while the searcher keeps working, so the pipeline looks healthy and silently verifies nothing.
4. **Live view** — the answer fragment is rendered (`format_answers.build_context` + `answer-body.html`) and pushed to the browser as an `answer` event, so the draft is visible while the searcher is still working and while the checkers run; the final state is `passed`, `exhausted` (last/empty answer shown), or `error`.

   An `AnswerWatcher` (one per round, in `based-answers.py`) polls `answers/<slug>.yml` every `ANSWER_POLL_SECONDS` for the lifetime of the searcher subprocess, so each `write_answer` call inside a round emits a fresh `answer` event and the answer updates in place as the agent iterates on it — several `answer` events per round is normal. It baselines the file already on disk at round start, so the previous round's leftover YAML is never shown as this round's output; it de-dupes on file content, and a rewrite caught half-written fails to render and is retried on the next poll. `search_loop()` emits once itself after the agent exits only if the watcher never emitted (`watcher.emitted`), so a round always displays the answer its checkers are checking. On the browser side, an `answer` arriving before the `deterministic` phase creates the "Citations verified verbatim against PDFs" check row in its pending state, so the draft is visibly unverified.

### Persistence & event stream

`citation-qa.db` (SQLite, WAL) in the CWD is the source of truth: `runs(run_id, question, slug, status running|passed|exhausted|error, created_at, finished_at, yaml_path)` and `events(id AUTOINCREMENT, run_id, event, data JSON, ts)`. `PipelineServer.emit()` inserts the event row and fans out to that run's connected WebSocket clients. `WS /ws/<run-id>` replays events after the client's last-seen id from the DB, then streams live — refresh, reconnect, and viewing after a restart are the same code path. A stream ends at a terminal `phase` event; server startup flips stale "running" runs to "error" and appends that terminal event.

**The event vocabulary is a coupling** shared by `based-answers.py` (emitters), the `events` table, and `run.html`'s client JS: `phase` (searching|deterministic|semantic|coherence|passed|exhausted|error, round), `agent-line` (agent searcher|semantic|coherence, ANSI-stripped line; semantic lines carry a `claim` index so the UI attaches them to that claim's stream; the context/feedback given to the searcher and the rubric given to each checker are emitted as stream lines too; a tool call emits `[tool] <name> <args>` when it starts, and — for `ECHOED_TOOLS`, currently just `verify_citations` — its **result** when it finishes, as a `── verify_citations result ──` block capped at `TOOL_RESULT_CHARS`, so the browser shows which citations failed and why at the moment the agent saw it. `pdf_search` is deliberately not echoed: an enumeration runs to 100 matches with snippets and would bury the reasoning the stream exists to explain. `pi_rpc.tool_result_text()` reads `tool_execution_end`; a line from the model's reasoning block carries `kind: "thinking"` and is rendered dimmed and italic — `run.html` therefore holds each stream as a list of `{line, kind}`, not as text), `claim-status` (index, claim, checking|pass|fail), `check-result` (deterministic|coherence, passed, output), `answer` (html fragment), `feedback` (round, text — persisted but not displayed; the UI shows the feedback via the next round's searcher stream), `tokens` (`by_agent` searcher|semantic|coherence → {input, output, cacheRead, cacheWrite, total, cost, calls}, plus `total`).

`tokens` comes from `TokenLedger`, fed by each `message_end`'s `AssistantMessage.usage` via `pi_rpc.message_usage()`. It is **cumulative for the whole question, not per round**, so `run.html` renders it outside the round navigation. Every event carries the complete running total rather than a delta, so a reconnect or a replay of a finished run rebuilds the same state from the last event alone. Note that summing `input` across a session counts the transcript once per API call — that double-counts tokens but not money, since it is exactly what the provider bills.

`run.html` buckets the ordered event stream into **rounds** client-side (`phase` events demarcate them): the page shows the latest round; earlier rounds' answers, citations, checks, and streams are reachable via the fixed round-navigation arrows at the bottom right. Streams are inline collapsibles: the searcher stream under the answer, a semantic-checker stream under each claim row, and the coherence stream under the coherence check row.

HTTP routes: `GET /` (question bar + run history + legacy `answers/*.yml` without run rows), `POST /ask`, `GET /run/<run-id>`, `GET /answer/<name>` (legacy YAML render), `GET /pdf/<basename>` (whitelist of registered PDFs only — pdf.js in the browser fetches these for page previews with highlight bboxes).

### The answer contract: the agent points at evidence, it does not type it

**The searcher cannot write citation text.** `write_answer` takes claims plus a *specification* of what backs each one, and `resolve-answer.py` materialises the text from the same extracted-text cache the verifier reads. A quote is therefore verbatim by construction, and a `search_result`'s `results` are a real rerun rather than a transcription. The entire class of "the agent believed it copied the snippet and hadn't" — which cost run `…-14` four verify calls on a single citation, and motivated the `>>>`/`<<<` paste-back machinery — cannot be authored any more.

The three citation types are the three ways to name evidence, and they take arguments in exactly the same way:

```jsonc
{"type": "quote",  "source": "file.pdf", "spans": [{"page": 14, "from": 12, "to": 19}]}
{"type": "search", "source": "file.pdf", "query": "plague"}
{"type": "regex",  "source": "file.pdf", "pattern": "Alderman [A-Z][a-z]+"}
```

**Line numbers are the addressing scheme, not chunks.** `chunk_paragraphs` splits on blank lines, and a datasheet has almost none: on the RP2040 fixture it yields 646 chunks for 640 pages, median 1999 chars — one chunk *is* a page, so a chunk ref would make every citation a 2000-char page dump. Lines are ~69 per page and roughly one clause each. `pdf-search.py` owns all of it (`page_text`, `page_lines`, `resolve_span`, `_chunk_start_lines`, `_span_of_window`) so the numbering the agent reads off a search hit is the numbering `write_answer` and the verifier resolve: `find_matches`/`find_distinct_matches` report the `[first, last]` lines their snippet covers, `cmd_search` prints them in the result header, and `cmd_get` prints every line with its number.

**A citation carries a *list* of spans, joined in order — that is how a cross-page quote works now.** The agent leaves out the lines it doesn't want, so a running header or footer interrupting a passage is simply skipped. `_split_match`, `MIN_CROSS_PAGE_CHARS` and the "at least ~20 chars must be on the stated page" rule exist only for legacy `text`-only answer files; nothing a live run writes goes near them. The citation's `page` is its first span's page, set by the resolver, never by the agent — as is `question`, which reaches the tool through `pi_env()`'s `ANSWER_QUESTION` for the same reason the slug already did.

**Nothing is written unless every citation resolves**, and *every* problem is reported in one pass — a half-resolved answer would go to the checkers as if the agent had meant it. A span past the end of a page, an unknown source, an invalid or too-broad pattern, and a span resolving below `MIN_CITATION_CHARS` are now authoring errors answered in milliseconds, not verifier round trips.

**`write_answer` returns the realised answer** (`readback()`), because the agent never typed the quote and this is its only look at what it actually cited: quote text in full, and each search/enumeration summarised to a count plus pages or the first `RESULTS_ECHO` match strings. Summarised deliberately — those snippets are already in context from the tool call that found them, and echoing an enumeration in full would double-carry 100 of them.

The on-disk YAML is unchanged apart from the added `spans`, so `format-answers.py`, the highlight bboxes, `answer-body.html`, the rubric blocks and legacy answer files all keep working:

```yaml
question: "..."
answers:
  - claim: "..."
    citations:
      - text: "<resolved from the spans>"
        page: 14
        source: "file.pdf"
        spans: [{page: 14, from: 12, to: 19}]
```

An empty `answers` list is the valid "unable to answer" output (and skips the semantic/coherence checks). Older answer files may still carry a `concatenation` field (once required to equal the claims joined with `". "`); nothing reads it anymore.

`verify-citations.py` dispatches on what the citation carries. A citation with `spans` goes to `check_spans()`, which **re-resolves rather than searches** — no leniency ladder, no cross-page case — and can only fail if the file was hand-edited or the source re-indexed. A `text`-only citation still gets the full match ladder (exact → whitespace-normalized → case-insensitive → hyphen-stripped → arrow-normalized → punctuation-stripped) plus `_verbatim_help()`/`_expand_verbatim()`, for legacy files.

A `search_result` citation (`type: search_result`, `source`, `query`, `results: [{page, text}, ...]`) proves absence ("the source never mentions X") or exhaustiveness ("this is the complete list") rather than quoting a passage. `verify-citations.py` reruns `query` against the cache (via `pdf_search.find_matches()`) and requires the claimed `results` to exactly match what comes back — by page, unbounded. Since the resolver produced those results from that same call, this now always agrees; it is kept because it still guards legacy files and because **the pattern-quality checks below live in the same branch and are not about transcription at all**. `based-answers.py`'s semantic rubric only adds the `SEARCH_RESULTS` block (and a query-adequacy critique — an obviously missing rephrasing/synonym should FAIL) to a claim that actually carries one; other claims see the unmodified rubric. `format-answers.py` renders these with no PDF preview/highlights, and deduplicates identical `(source, query, results)` search_result citations across claims into one reference — a separate exact-match pass from the fuzzy quote-overlap merge normal citations get.

**`search_regex` returns distinct matched *strings*, and a pattern written as if it were a prose search silently means something else.** `regex_notes()` answers the two misreadings that cost `…pin-directions-of-a-pio-11` most of its rounds, in the tool's own output where the agent will read them: a pattern with no variable part (`pio_sm_get_`) can only match itself, so a document full of such functions dedupes to a single row whose "name" is the prefix — read literally, a family of one; and `.` matches spaces, so `pio.*pindir` runs from the first `pio` on a page to the last `pindir` and the distinct strings are whole paragraphs, at which point dedup is meaningless and the enumeration can't be cited. Long match strings are also truncated in the result header (`MATCH_LABEL_CHARS`) so a greedy pattern doesn't bury its own output.

The third and worst one is `_truncation_diagnostic()`, from run `-12`: a **requirement at the end** of an otherwise-correct pattern. `pio[a-z_]*pindir[a-z_]*\(` finds one function where the same pattern without `\(` finds three, because the reference section renders `name (args)` with a space and only code listings render `name(`. Nothing errors — the agent gets a shorter list that still looks like an answer, and read the drop as the tool truncating output. So the tool reports the longest prefix of the pattern that finds **strictly more**, with what follows each extra match verbatim in the text (`' (PIO pio, uint sm'` — the space, right there). Two rules keep it from crying wolf, both learned from false positives on real patterns: extra matches must be **novel**, not case-insensitive fragments of hits the pattern already had (else `pio[a-z_]*` "finds" a bare `pio` inside every existing hit), and the dropped tail must be **structural** — escapes, classes, quantifiers, anchors — never a search term (`_tail_is_structural()`; dropping `dir[a-z_]*` asks a different question and always finds more). A genuine absence therefore gets no note at all, which is the point: a false "your pattern is broken" would send the agent hunting for something that isn't there. The suggested repairs are tested to produce the enumeration the agent was actually after.

A `search_result` citation may also set `mode: regex`, for claims that rule out a whole *family* of possible names rather than one literal phrase (e.g. "no `pio_sm_*` function reads back enabled state") — a handful of guessed literal queries can never be exhaustive. The agent asks for one with `{"type": "regex", "pattern": ...}`; in the resolved YAML, `query` is that full Python regex and each result is `{match, page, text}` — `pdf_search.find_distinct_matches()` (agent-facing as the `search_regex` tool action) enumerates every *distinct* string the pattern matches anywhere in the doc, deduplicated, capped (currently 100, tunable) so a too-broad pattern fails loudly rather than silently truncating (a truncated enumeration could hide the exact symbol that disproves the claim). Its per-match snippets are half the width of a normal search's (150 vs 300 chars): 100 of them are carried twice, once in the tool result and again in the YAML the `ENUMERATION` block quotes back — which is why `readback()` summarises rather than echoing a third copy. `verify-citations.py`'s regex branch checks each claimed result's `text` is on-page and its `match` is actually produced by re-running the pattern against that same (normalized) text, then requires the claimed match-string *set* to exactly equal an independent rerun's.

**That set comparison has one blind spot, closed by `_check_enumeration_covers_digits()`.** An identifier-family pattern written `pio_sm_set_[a-z][a-z_]*` cannot match a digit, so `pio_sm_set_pindirs_with_mask64` comes back truncated to `…_with_mask` — or, if the digitless class sits before a required suffix like `\s*\(`, drops out of the enumeration entirely. Neither shows up above, because the rerun uses the *same* crippled pattern and both sides agree: an exhaustiveness claim whose list is missing exactly the name that would refute it verifies clean. So `_digit_permissive_variant()` rebuilds the pattern with `0-9` added to every character class that can't already match a digit (skipping negated classes and ones already carrying `\d`/`\w`), re-enumerates, and fails the citation with the concrete missing names if the permissive pattern finds more. It fails on **evidence, not syntax** — a digitless class that changes nothing about the result passes, and a repaired pattern that turns out invalid or too broad proves nothing about the original and is ignored.

**The same blind spot has a second form, closed by `_check_enumeration_is_general()`.** A pattern with no character class, quantifier or wildcard — `pio_sm_set_pindirs_with_mask64|pio_sm_set_pindirs_with_mask|pio_sm_set_consecutive_pindirs`, the round-5 citation of `…pin-directions-of-a-pio-10` — is an alternation of literals offered as proof that those literals are the only members of a family, when it could never have matched a fourth. The set comparison agrees with itself as before, and the digit repair finds no class to widen. `spelled_out_literals()` (in `pdf-search.py`, so the search tool and the verifier share one definition) returns the branches when the pattern is non-generative, or `None` the moment it sees `[`, `.`, `*`, `+`, `?`, `{` or `\d`/`\w`/`\s` and friends. Evidence again, not syntax: a spelled-out pattern that finds **nothing** has ruled out those literals and made no exhaustiveness argument, so it passes; one that finds them is failed. Nothing about this rule is in `citation-searcher.md` on purpose — the verifier's failure message is what teaches it, at the point of use. The rubric's `ENUMERATION` block shows the checker the complete, verified match list and asks it to judge whether that list (plus SOURCE_TEXTS) supports CLAIM, and whether the pattern itself covers the right family — ordinary LLM judgment, same as the rest of this checker's reasoning.

### Key couplings to keep in sync

- A run's filesystem slug is always its `run_id` (the possibly `-N`-suffixed id `PipelineServer.create_run()` assigns once, up front, to dedupe re-asks of an identical question) — `run_worker()` never re-derives a slug from the question text. `open_search_session()` passes that slug to the pi subprocess via `pi_env()`'s `ANSWER_SLUG`; `tools/write-answer.ts` reads it from `process.env`, hands it to `resolve-answer.py --slug`, and that unconditionally overwrites `answers/<slug>.yml` — the searcher agent itself never sees or handles a slug, and the tool never derives its own or bumps `-N`, so a run's answer file never sprawls across rounds. `find_yaml()` still tolerates a `<slug>-N.yml` match (max by mtime) for legacy multi-file directories predating this, but a live run only ever produces the exact `<slug>.yml`. `derive_slug()` is used only to seed `create_run()`'s dedup check and as the env-var fallback if `run_id` is ever absent.
- The agent's permissions are set **at the call site**, not in frontmatter: `--no-builtin-tools` plus `--tools` in `build_command()` is the whole allowlist. `citation-searcher.md` is now only a system prompt — any tool/permission frontmatter left in it is inert. (The old pipeline counted denied-permission lines in agent output for retry diagnostics; with no built-in tools to deny, there is nothing to count.)
- `pdf-search.py`'s cache format (`chunks` with `page`/`text`, plus lazily-added `pages_data` with span bboxes) is consumed by `verify-citations.py` and `format-answers.py` directly.
- `answer-body.html` is the shared answer markup: included by `answer-template.html` (static pages via `format-answers.py`, which remains a standalone CLI renderer writing `answer-pages/<slug>.html`) and rendered as the `answer` event fragment by `PipelineServer.render_answer_fragment()`. Refs carry a `pdf_url` (`file://…` for static output, `/pdf/<basename>` in the app) — `build_context(yaml_path, pdf_url_for=...)` controls it.
- `run.html`'s status bar takes `max_rounds` from `MAX_ROUNDS` in `based-answers.py` (via the server); never hardcode the round count in templates.
